# Resonance2 v2

## Что это такое

Это сервис для совместного прослушивания одного аудиоисточника в комнате.

Сервер не передает музыку сам.
Он только хранит состояние комнаты и говорит всем участникам:

- какой сейчас трек
- играет он или стоит на паузе
- с какой позиции нужно играть
- в какой момент по времени сервера это нужно сделать

Сам звук каждый браузер загружает напрямую по `Track URL`.

## Как это работает простыми словами

1. Один пользователь создает комнату.
2. Другие заходят в эту же комнату по `Room ID` или по ссылке.
3. Хост задает ссылку на аудио.
4. Когда кто-то нажимает `Play`, фронт отправляет команду на сервер.
5. Сервер не говорит "играй прямо сейчас", а планирует запуск немного вперед по своему времени.
6. Все клиенты знают время сервера и стараются стартовать одновременно.
7. Если у кого-то плеер начинает немного убегать вперед или назад, фронт мягко подправляет скорость через `playbackRate`.
8. Если рассинхрон стал слишком большим, фронт делает точный `seek` на нужную позицию.

## Зачем нужна кнопка "Готов слушать"

Браузеры часто запрещают запускать звук без действия пользователя.

Поэтому на каждом устройстве пользователь должен один раз нажать кнопку `Готов слушать`.
После этого браузер обычно разрешает локальное воспроизведение.

## Что хранит сервер

Для комнаты сервер хранит:

- `RoomId`
- `TrackUrl`
- текущее состояние `play / pause`
- текущую позицию трека
- версию состояния
- отложенную команду, если нужно выполнить `play` или `seek` в будущем

## Что делает фронт

Фронт в `v2/app.js` делает несколько важных вещей:

### 1. Подключается к SignalR hub

По умолчанию фронт подключается сюда:

`https://api.cloud-platform.pro/resonance/musicHubV2`

### 2. Синхронизирует часы с сервером

Фронт несколько раз вызывает метод `SyncClock`.
По ответам он считает:

- задержку сети
- разницу между временем клиента и временем сервера

Это нужно, чтобы все устройства ориентировались на одно и то же время.

### 3. Получает состояние комнаты

Когда пользователь входит в комнату, сервер отправляет текущее состояние комнаты.

### 4. Планирует выполнение команд

Если сервер прислал команду вроде:

- `play` в 15:30:10.200
- `seek` в 15:30:15.450

то фронт ждет именно этого момента по серверному времени и выполняет действие локально.

### 5. Корректирует drift

Если локальный плеер чуть-чуть уехал, фронт:

- не делает резкий прыжок сразу
- сначала немного меняет `playbackRate`

Если рассинхрон уже заметный, тогда делает жесткий `seek`.

## Какие методы вызывает фронт у хаба

Фронт вызывает такие методы SignalR:

### `CreateRoom()`

Создает новую комнату.
Возвращает `Room ID`.

### `JoinRoom(roomId)`

Добавляет пользователя в комнату.
После этого сервер отправляет текущее состояние комнаты.

### `SetTrack(roomId, url)`

Устанавливает ссылку на аудио для комнаты.

### `Play(roomId, startTime)`

Говорит серверу, что нужно начать воспроизведение с позиции `startTime`.
Сервер планирует запуск немного вперед по времени.

### `Pause(roomId)`

Ставит комнату на паузу.

### `Seek(roomId, time)`

Меняет позицию трека.
Если трек сейчас играет, сервер может запланировать `seek` немного вперед, чтобы все устройства прыгнули почти одновременно.

### `SyncClock(clientSendUnixMs)`

Нужен для синхронизации часов клиента с сервером.

## Какие события сервер отправляет на фронт

### `RoomUpdated`

Приходит после входа в комнату.
Содержит полное текущее состояние.

### `StateChanged`

Приходит после изменений в комнате:

- новый трек
- play
- pause
- seek

Фронт обновляет локальный плеер по этим данным.

## Что важно понимать

- Это не стриминг-сервис.
- Сервер не раздает аудио.
- Все клиенты сами загружают один и тот же аудио URL.
- Обычные ссылки на YouTube или Spotify сюда не подходят, если это не прямой аудиопоток.

## Типичный сценарий

1. Пользователь открывает `v2/index.html`.
2. Создает комнату.
3. Вставляет прямую ссылку на аудио.
4. Другой пользователь заходит по ссылке.
5. На втором устройстве нажимают `Готов слушать`.
6. Хост запускает трек.
7. Все устройства стараются начать воспроизведение в один и тот же момент по времени сервера.

## Ограничения

Даже в `v2` нельзя обещать идеально одинаковый звук до миллисекунды на любых устройствах, потому что влияют:

- браузер
- декодер аудио
- буферизация
- autoplay policy
- качество сети

Но `v2` работает заметно точнее, чем простой вариант с периодическим `drift correction`.
